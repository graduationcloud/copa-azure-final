## `[AUTO-DECISION]` — as 2 pendências de design da ADE-008, resolvidas aqui

A ADE-008 deixou explicitamente 2 decisões "a confirmar no draft da story" (§Verificação na fonte da verdade + Decisões pendentes do épico). Resolvo ambas agora, com recomendação e justificativa — não deixo vago:

### (a) Forma concreta da notificação inline: HTTP mock vs só log → **LOG ESTRUTURADO**

**Decisão:** a notificação inline é **um `ILogger.LogInformation`**, sem nenhuma chamada `HttpClient`.

**Por quê (justificativa):**
- **Ambas as formas honram a decisão do owner** ("mock HTTP/log") — a ADE-008 é explícita que as duas são válidas. Entre as duas, log é estritamente mais simples.
- **Elimina a ÚLTIMA dependência de rede/configuração do pós-compra.** Um HTTP mock (ex.: `httpbin.org/post`, usado no F4 antigo para o AI Agent) ainda exigiria uma URL configurável (novo App Setting), timeout, tratamento de falha de rede — exatamente a cerimônia operacional que a ADE-008 existe para eliminar. Ninguém mais inspeciona uma UI do n8n para "ver o webhook chegar"; o valor pedagógico de uma chamada HTTP de fato desapareceu junto com o n8n.
- **É trivialmente observável.** Um log dentro do `BeginScope` já correlacionado aparece no App Insights exatamente como qualquer outro hop — e é exatamente isso que o F6 consome (`AppInsightsFlowEventRepository` consulta `traces`, não faz HTTP). Um log estruturado é **mais alinhado** ao mecanismo real do F6 do que uma chamada HTTP seria.
- **Reduz superfície de teste.** Testar "um log foi emitido com este correlationId" é mais simples e mais rápido (sem `HttpMessageHandler` stub) do que testar uma chamada HTTP mockada — reduzindo o atrito de manter os testes herdados de `N8nWebhookNotifierTests`.
- **Alternativa registrada (não escolhida):** se o `@architect`/owner quiser reintroduzir um efeito HTTP visível no futuro (ex.: para uma demo mais "palpável"), isso pode ser adicionado depois como um `IHttpClientFactory` opcional atrás de um App Setting vazio-por-padrão (mesmo padrão fail-open do `N8nWebhookNotifier` antigo) — mas **não é o default desta story**.

### (b) F6 = 5 nós vs manter um 6º nó `NOTIFICATION_SENT` → **5 NÓS (default da ADE-008)**

**Decisão:** `FlowEventType` fica com **5 membros** (0-4). A notificação pós-compra fica **implícita**, "dobrada" dentro do nó `FUNCTION_CONSUMER_DONE` (índice 3) — não ganha nó próprio.

**Por quê (justificativa):**
- **É o default explícito da ADE-008** ("o default desta decisão é 5 nós") — a opção de um 6º nó foi **deferida ao `@sm`/`@dev`**, não recomendada.
- **Consistência com a razão de ser da simplificação.** Reintroduzir um nó dedicado para uma notificação que agora é só um log dentro do mesmo hop do Consumer recriaria visualmente a impressão de uma "etapa extra" que a remoção do n8n justamente eliminou — indo contra a lição didática pretendida ("por que o pós-compra ficou mais simples").
- **Trade-off aceito é honesto e já documentado na ADE-008** ("a notificação pós-compra fica invisível na animação... pequena perda didática, aceita em nome da simplificação"). A Story 3.4 (F6 lab) inclui uma AC específica para tornar essa perda **didaticamente explícita** (o aluno inspeciona o payload do nó Consumer e vê a notificação ali dentro), compensando a perda de um nó visual com uma explicação guiada.
- **Alternativa registrada (não escolhida):** um 6º nó `NOTIFICATION_SENT` alimentado pelo trace da notificação inline — tecnicamente trivial de adicionar depois (é só reintroduzir um membro de enum + um branch de classificação), portanto **não é uma porta fechada**, apenas não é o default.
